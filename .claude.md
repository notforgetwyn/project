# Python + Pygame 俄罗斯方块项目蓝图

## 1. 协作规则

本项目按以下规则推进：

1. 先输出简洁但专业的需求分析文档。
2. 再输出项目架构设计，包括项目目录结构、项目架构设计、页面设计、UI 设计、模块划分、类设计、数据流设计。
3. 然后把项目拆成多个迭代版本：MVP、菜单、设置、存档、优化。
4. 每次只完成一个小阶段，不一次性写完整个项目。
5. 每个阶段都先说明目标、实现思路、涉及文件，再给代码。
6. 所有代码必须可运行、结构清晰、便于扩展，禁止把所有逻辑写进一个文件。
7. 优先使用面向对象设计，加入必要注释，但不过度冗余。
8. 配置和存档统一使用 JSON 文件。
9. 写代码时考虑异常处理、边界情况和后续扩展。
10. 每完成一个阶段，都给出下一阶段建议。

额外约束：

- 先从最小可运行版本开始：方块下落、旋转、消除、行计分、游戏结束。
- 然后再逐步加入主菜单、设置、存档。
- 用户说"继续下一步"后，才进入下一个阶段。
- 用户说"重构"后，按工程化方式优化当前结构。
- 用户说"解释"后，用适合初学者的方式解释代码。

## 2. 需求分析

### 2.1 项目目标

开发一个基于 Python + Pygame 的桌面端俄罗斯方块游戏，具备可玩性、可维护性和可扩展性，并按照工程化迭代方式逐步建设。

### 2.2 核心功能需求

1. 俄罗斯方块支持左、右移动。
2. 俄罗斯方块支持加速下落。
3. 俄罗斯方块支持旋转。
4. 方块自然下落并在触底后固定。
5. 支持完整行消除并计分。
6. 游戏过程中实时显示分数和等级。
7. 游戏启动后先进入开始界面，而不是直接开始。
8. 开始界面支持：
   - 开始游戏
   - 继续游戏
   - 设置
   - 退出游戏
9. 支持游戏存档与读档。
10. 设置页面支持调整消行速度、难度，并持久化保存。
11. 整体架构支持后续扩展，例如消行特效、音效、方块皮肤、排行榜等功能。

### 2.3 非功能需求

1. 代码结构清晰，模块职责明确。
2. 采用面向对象设计。
3. 配置与存档使用 JSON 存储。
4. 具备基本异常处理能力，例如配置文件缺失、存档损坏、非法数据回退默认值。
5. 便于测试和后续重构。

### 2.4 边界与风险

1. 方块旋转时需要检测边界，避免超出游戏区域。
2. 方块生成时如果空间不足，判定游戏结束。
3. 配置文件或存档文件不存在时，需要自动创建或回退默认值。
4. 读档时需要校验数据合法性，避免程序崩溃。
5. 等级提升时速度变化需要设置合理上下限，避免不可玩。
6. 消行时需要处理多行同时消除的计分。

#### 分层说明

1. UI 展示层
   - 负责页面展示、按钮交互、文本显示和 HUD 渲染。
   - 不直接持久化数据，不承担底层文件读写职责。

2. 应用控制层
   - 负责程序启动、Pygame 初始化、主循环管理、事件分发和场景切换。
   - 连接 UI 展示层与业务逻辑层，是运行时调度中心。

3. 业务逻辑层
   - 负责方块移动、旋转、碰撞检测、触底固定、行消除、等级判定等核心规则。
   - 尽量不依赖具体页面，以保证后续可测试和可复用。

4. 配置与存档服务层
   - 负责设置和存档的读取、保存、校验、默认值补全和异常兜底。
   - 向上为场景和应用层提供统一的数据访问接口。

5. 文件管理层
   - 负责组织配置目录、存档目录、资源目录和文档目录。
   - 屏蔽具体文件路径，避免业务代码散落硬编码路径。

6. 数据存储层
   - 当前阶段采用 JSON，适合设置、存档和轻量数据持久化。
   - 后续如果出现排行榜、成就或复杂用户数据，再考虑升级到 SQLite。

7. 日志与诊断预留层
   - 当前版本先预留接口和位置。
   - 后续可加入日志记录、错误诊断和调试开关，提升可维护性。

#### 架构设计原则

1. UI 与业务逻辑解耦，避免把游戏规则直接写死在页面渲染代码中。
2. 配置与存档统一通过服务层访问，避免后续功能增加时重复读写 JSON。
3. 主循环只负责调度，不承载过多业务细节。
4. 各层职责单一，便于后续加入暂停页、音效系统、排行榜和主题皮肤。

### 3.3 页面与 UI 对应设计

#### 启动流程

1. 启动程序
2. 进入主菜单页
3. 从主菜单进入开始游戏 / 继续游戏 / 设置 / 退出

#### Page 1: 主菜单页 `MenuScene`

页面目标：
- 作为程序统一入口
- 引导玩家开始新游戏或进入其他功能页

UI 组成：
- 顶部标题：`Tetris`
- 中部菜单项：
  - 开始游戏
  - 继续游戏
  - 设置
  - 退出游戏
- 底部操作提示：方向键或 `W/S` 选择，`Enter` 确认

布局设计：
- 标题位于页面上方居中
- 菜单按钮在垂直方向居中排列
- 当前选中项使用高亮色
- 底部显示轻量提示文本

交互设计：
- `Up/Down` 或 `W/S` 切换选项
- `Enter/Space` 执行当前选项
- `ESC` 退出程序

视觉设计：
- 背景：深灰/黑色
- 标题：白色大字号
- 普通菜单项：白色
- 选中菜单项：蓝色高亮

#### Page 2: 游戏页 `GameplayScene`

页面目标：
- 承载核心玩法
- 展示实时游戏状态

UI 组成：
- 游戏面板（10列 x 20行网格）
- 下一个方块预览区域
- 顶部 HUD：
  - 当前分数
  - 当前等级
  - 消行数
- 操作提示

布局设计：
- 游戏面板居中
- 右侧显示下一个方块预览
- 左侧显示分数、等级、消行数
- 游戏结束时在中间显示结束信息层

交互设计：
- `Left/Right` 或 `A/D` 控制左右移动
- `Up` 或 `W` 控制旋转
- `Down` 或 `S` 控制加速下落
- `Space` 控制硬降（直接落到底）
- `ESC` 返回主菜单
- 游戏结束后按 `R` 重新开始

视觉设计：
- 背景：深色
- 网格线：低对比灰色
- 方块：不同颜色对应不同形状（I=青色、O=黄色、T=紫色、S=绿色、Z=红色、J=蓝色、L=橙色）
- 空格子：深色背景
- HUD 文本：白色
- 结束提示：红色标题 + 白色说明

#### Page 3: 设置页 `SettingsScene`

页面目标：
- 调整游戏难度和速度
- 保存用户配置

UI 组成：
- 页面标题：`Settings`

布局设计：
- 标题顶部居中
- 中部展示难度调节区域
- 底部展示保存和返回按钮

交互设计：
- 左右键或按钮调整难度
- `Enter` 保存当前配置
- `ESC` 返回菜单

视觉设计：
- 与主菜单保持统一风格
- 当前选中控件高亮
- 难度值用更醒目的字号显示

#### Page 4: 继续游戏页 `Continue / Load`

页面目标：
- 从存档恢复游戏
- 在无存档时提供清晰反馈

UI 组成：
- 页面标题：`Continue Game`
- 存档状态文本：
  - 有存档：显示可继续
  - 无存档：显示暂无存档

交互设计：
- `R` 重新开始
- `ESC` 返回菜单

视觉设计：
- 结束标题红色突出失败状态
- 得分与操作说明使用白色文本

### 3.4 全局 UI 规范

为保证每个页面风格统一，定义以下通用规范：

1. 颜色规范
   - 背景：深灰/黑色
   - 主文本：白色
   - 强调色：蓝色
   - 警告/结束：红色
   - 方块颜色：
     - I 方块：青色
     - O 方块：黄色
     - T 方块：紫色
     - S 方块：绿色
     - Z 方块：红色
     - J 方块：蓝色
     - L 方块：橙色
2. 布局规范
   - 页面主内容尽量居中
   - HUD 固定角落显示，不遮挡核心玩法
   - 菜单类页面采用垂直排列
3. 组件规范
   - 文本组件：标题、正文、提示三种字号
   - 按钮组件：默认态、选中态、禁用态
   - 提示组件：用于无存档、保存成功、错误信息
4. 扩展预留
   - 后续可加入图标、动效、音效和主题皮肤

#### 类职责说明

1. `App`
   - 负责程序生命周期管理。
   - 持有当前场景，并负责场景切换。

2. `BaseScene`
   - 定义统一场景接口，保证所有页面接入方式一致。

3. `MenuScene`
   - 负责主菜单展示和菜单选择逻辑。

4. `GameplayScene`
   - 负责单局游戏流程，协调方块生成、下落、旋转、消除和结束判定。

5. `SettingsScene`
   - 负责难度设置展示、修改和保存。

6. `ContinueScene`
   - 负责读取存档状态并决定是否恢复游戏。

7. `GameOverScene`
   - 负责展示结算信息并提供后续操作入口。

8. `Tetromino`
   - 负责方块形状、位置、旋转状态和移动逻辑。

9. `TetrominoFactory`
   - 负责根据七种标准方块模板创建随机方块。

10. `Board`
    - 负责游戏区域状态、方块锁定、行消除判定和游戏结束判定。

11. `GameSession / GameData`
    - 负责保存当前局所需的运行数据和可持久化数据结构。

12. `FileManager / SettingsService / SaveService`
    - 负责 JSON 数据读写、异常处理和默认值兜底。

### 3.7 数据流设计

```text
用户输入
   ↓
当前场景接收事件
   ↓
应用控制层分发动作
   ↓
业务逻辑层更新游戏状态
   ↓
配置/存档服务层按需读写 JSON
   ↓
场景重新渲染 UI
```

#### 运行时数据流

```text
键盘输入
   ↓
GameplayScene.handle_event()
   ↓
Tetromino.move() / rotate()
   ↓
Board.is_collision() 检测碰撞
   ↓
触底后 Board.lock_tetromino()
   ↓
Board.clear_lines() 消除计分
   ↓
GameplayScene.render()
   ↓
屏幕显示最新游戏状态
```

#### 配置数据流

```text
程序启动
   ↓
SettingsService.load_settings()
   ↓
读取 config/settings.json
   ↓
非法或缺失则回退默认值
   ↓
应用到游戏运行参数
   ↓
用户在设置页修改难度
   ↓
SettingsService.save_settings()
   ↓
写回 settings.json
```

#### 存档数据流

```text
游戏运行中
   ↓
GameSession / GameData 生成当前状态
   ↓
SaveService.save_game()
   ↓
写入 data/save.json
   ↓
主菜单点击继续游戏
   ↓
SaveService.load_save()
   ↓
恢复棋盘状态、当前方块、分数、等级等状态
   ↓
GameplayScene 根据存档恢复运行
```

#### 数据流设计原则

1. 页面层只消费数据和触发动作，不直接操作底层文件。
2. 业务状态变化先发生在内存对象，再通过服务层决定是否持久化。
3. JSON 文件始终通过统一入口读写，避免多个模块各自维护格式。
4. 数据流方向尽量单向，降低后续调试和扩展成本。

#### 场景 1: 启动场景 `StartScene`

退出条件：初始化完成

转向：MenuScene

#### 场景 2: 主菜单场景 `MenuScene`

前置条件：无

进入条件：程序启动、任意场景返回

场景流程：
1. 显示主菜单界面
2. 玩家选择菜单项
3. 执行对应操作

菜单项：
- 开始游戏：清空存档，进入 GameplayScene
- 继续游戏：检查存档，进入 ContinueScene
- 设置：进入 SettingsScene
- 退出游戏：退出程序

退出条件：选择菜单项后跳转

转向：GameplayScene / ContinueScene / SettingsScene / EXIT

#### 场景 3: 游戏场景 `GameplayScene`

前置条件：无

进入条件：MenuScene 选择开始游戏 / ContinueScene 选择继续

场景流程：
1. 初始化游戏板和方块生成器
2. 生成第一个方块
3. 进入游戏主循环：
   - 接收玩家输入（左/右/旋转/加速/硬降）
   - 更新方块位置和状态
   - 检测碰撞和消行
   - 更新分数和等级
4. 检测游戏结束条件
5. 游戏结束后显示结算

退出条件：
- 游戏结束按 ESC：保存存档，返回 MenuScene
- 游戏结束按 R：重新开始
- 游戏进行中 ESC：保存存档，返回 MenuScene

转向：MenuScene / GAME_OVER

#### 场景 4: 继续游戏场景 `ContinueScene`

前置条件：无

进入条件：MenuScene 选择继续游戏

场景流程：
1. 检查存档文件是否存在
2. 读取并校验存档数据
3. 判断存档是否有效：
   - 有效存档：显示继续确认
   - 无效/损坏存档：显示提示，返回 MenuScene

退出条件：
- 选择继续：恢复存档数据，进入 GameplayScene
- 选择返回：返回 MenuScene

转向：GameplayScene / MenuScene

#### 场景 5: 设置场景 `SettingsScene`

前置条件：无

进入条件：MenuScene 选择设置

场景流程：
1. 加载当前设置
2. 显示设置界面
3. 玩家调整设置项（难度等级）
4. 玩家选择保存或返回

退出条件：
- 保存设置：写入配置文件，返回 MenuScene
- 直接返回：不保存，返回 MenuScene

转向：MenuScene

#### 场景 6: 游戏结算场景 `GameOverScene`

前置条件：游戏结束

进入条件：GameplayScene 游戏结束

场景流程：
1. 显示游戏结束画面
2. 显示最终得分和消行数
3. 等待玩家选择操作

操作选项：
- 重新开始：重置游戏，进入新的 GameplayScene
- 返回主菜单：返回 MenuScene

退出条件：玩家选择操作后跳转

转向：GameplayScene（重启）/ MenuScene

#### 场景 7: 退出场景 `Exit`

前置条件：无

进入条件：MenuScene 选择退出游戏

场景流程：
1. 关闭所有资源
2. 退出程序

#### 核心业务场景详细说明

##### 场景 A: 正常游戏流程

1. 玩家从菜单选择"开始游戏"
2. 系统清空旧存档（如有）
3. 系统初始化空白棋盘
4. 系统生成第一个随机方块
5. 方块开始自然下落
6. 玩家通过方向键控制方块移动和旋转
7. 方块触底后自动固定
8. 系统检测并消除完整行
9. 系统生成下一个方块
10. 循环步骤 6-9 直到游戏结束
11. 玩家查看得分，选择重新开始或返回菜单

##### 场景 B: 存档继续流程

1. 玩家从菜单选择"继续游戏"
2. 系统检查存档文件
3. 系统读取并校验存档数据
4. 存档有效则恢复游戏状态
5. 玩家继续游戏

##### 场景 C: 设置调整流程

1. 玩家从菜单选择"设置"
2. 系统加载当前设置
3. 玩家调整难度等级
4. 玩家选择保存
5. 系统写入配置文件
6. 返回主菜单

##### 场景 D: 异常处理流程

1. 存档文件不存在：提示无存档，返回菜单
2. 存档数据损坏：使用默认值，提示存档已损坏
3. 配置文件缺失：创建默认配置文件
4. 方块无法生成（空间不足）：触发游戏结束
